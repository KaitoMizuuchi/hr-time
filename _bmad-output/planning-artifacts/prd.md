# Product Requirements Document - hr-time

**Author:** かいと
**Date:** 2026-03-13

## Executive Summary

hr-timeは、日々の稼働時間の記録・分類・集計を自動化するタイムボクシング式タスク管理Webアプリである。ターゲットユーザーは、毎日複数タスクの稼働時間をExcelにまとめて提出する必要があるナレッジワーカー。「何をやったか思い出す」「分類を考える」「時間を集計する」という3つの苦痛を、計画編集型のタイムラインUIで根本から解消する。初期は個人利用、将来的にチームへの展開を見据える。

### What Makes This Special

タイマー型ではなく**計画編集型**のアプローチを採用。ユーザーはタイムラインにタスクブロックを配置して計画を立て、実際とズレたら修正するだけ。「計画 ＝ 記録」というパラダイムにより、記録のための記録が不要になる。プロジェクト配下でタスクを作成すればタグが自動付与され、分類の手間もゼロ。月次集計とExcel出力はワンクリックで完了する。

## Project Classification

- **Project Type:** Web App（SPA）
- **Domain:** 業務効率化・生産性ツール
- **Complexity:** Low（標準的なCRUD + レポート出力）
- **Project Context:** Greenfield（新規開発）

## Success Criteria

### User Success

- 一日の稼働報告が**数クリック**で完了できる（Excel手入力からの脱却）
- タスク管理と時間記録が一体化し、別々のツールを使い分ける必要がない
- 「何をやったか」「どの分類か」を思い出す必要がない — タイムラインに置いた時点で記録済み
- 月次集計が手作業ゼロで完了する

### Business Success

- BMADワークフローによる開発速度の検証（企画→実装までの所要時間を計測）
- 開発者自身が毎日使い続けられるクオリティの達成

### Technical Success

- SPAとして快適に動作するレスポンス速度
- 15分単位のタイムラインUIがスムーズに操作できるインタラクション品質
- データの整合性（計画/実績の自動判定、タグ別集計の正確性）

### Measurable Outcomes

- 日次の稼働報告作業: 現状15-30分 → **3分以内**
- 月次集計作業: 現状数時間 → **ワンクリック（数秒）**
- アプリ定着: 開発者自身が**2週間以上継続利用**

## User Journeys

### Journey 1: かいと — 朝の計画から終業確認まで（プライマリユーザー・成功パス）

**ペルソナ:** かいと、エンジニア。複数案件を並行して担当。月次で稼働時間をExcelで提出する必要がある。

**Opening Scene:**
朝10時、かいとはPCを開く。今日は案件Aの設計レビュー、案件Bの定例会議、案件Cの実装がある。hr-timeを開くと、左のTodoリストにプロジェクトごとのタスクが並んでいる。「今日はこれとこれをやろう」と、タスクの横にある「今日の予定にする」ボタンを押し、ざっくり30分単位で時間枠を選ぶ。3-4個のタスクをタイムラインに配置。昼休憩も休憩トグルで入れておく。所要時間2-3分。

**Rising Action:**
午前中、案件Aの設計レビューが予定より30分延びた。かいとはタイムライン上のブロックをクリックして時間を修正。案件Bの定例は予定通り。午後、急な依頼が入り案件Cの実装を中断して対応。Todoリストから新しいタスクを「今日の予定にする」でタイムラインに追加。案件Aの設計は昨日から継続しており、同じタスクに対して今日のタイムブロックが追加される。

**Climax:**
夕方19時、終業時間。かいとはタイムラインを見る。今日一日の作業が全てブロックとして並んでいる。ズレた部分だけ微調整して確認完了。

**Resolution:**
日々の記録はこれだけ。何をやったか思い出す必要がない。時間の計算も不要。月末にまとめてExcel出力すればいい安心感がある。

---

### Journey 2: かいと — 計画なしで後から実績入力（エッジケース）

**Opening Scene:**
月曜朝、かいとは急ぎの障害対応に追われ、計画を立てる余裕がないままバタバタと一日が始まる。

**Rising Action:**
午前中は障害対応、午後は定例会議と通常タスク。hr-timeを開く暇がなかった。

**Climax:**
夕方、ようやく落ち着いてhr-timeを開く。タイムラインは空白。しかし計画は必須ではない。かいとはTodoリストからタスクを選び、過去の時間枠に実績としてブロックを配置していく。現在時刻より前なので自動的に「実績」として記録される。

**Resolution:**
計画がなくても、タイムラインにブロックを並べるだけで記録が完了。従来のExcel手入力より遥かに楽。ただし「思い出す」作業は残るため、やはり朝に計画を立てた方が楽だと実感する。

---

### Journey 3: かいと — 月末集計レポート作成

**Opening Scene:**
月末最終日。かいとは今月の稼働実績を案件別にまとめて提出する必要がある。以前は一日ずつExcelを見返して集計していた。

**Rising Action:**
hr-timeの月次ダッシュボードを開く。日別×プロジェクト別の稼働時間一覧が表示される。「3月5日：案件A 3時間、案件B 2時間」のように、各日の作業内容と時間が一目瞭然。プロジェクトタグ別、作業種別タグ別の月間合計も自動集計されている。

**Climax:**
「Excel出力」ボタンを押す。提出用フォーマットに整形されたExcelファイルがダウンロードされる。

**Resolution:**
以前は数時間かかっていた月末集計が数秒で完了。

## Web App Specific Requirements

### Project-Type Overview

SPA（シングルページアプリケーション）として構築。React + Viteによるクライアントサイドレンダリング。SEO不要のため、SSRは不要。

### Technical Architecture Considerations

- **SPA構成**: React + Vite、クライアントサイドレンダリング
- **ブラウザサポート**: Chrome, Safari, Edge, Firefox（最新2バージョン）
- **SEO**: 不要（個人/社内利用ツール）
- **リアルタイム同期**: MVP時点では不要。TanStack Queryによる楽観的更新で即時UI反映
- **アクセシビリティ**: WCAG Level A準拠（基本的なキーボード操作、スクリーンリーダー対応）

### Responsive Design

- **プライマリ**: デスクトップ（2カラムUI）
- **セカンダリ**: タブレット/モバイル対応は後期検討（MVPはデスクトップ優先）

### Implementation Considerations

- TanStack Queryによるサーバー状態管理と楽観的更新
- react-hook-form + zodによるフォームバリデーション
- Tailwind CSS + shadcn/uiによるコンポーネント設計
- Biomeによるコード品質管理
- vitestによるテスト

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** Problem-Solving MVP — 「毎日の稼働記録・月次集計の苦痛」という特定の問題を最小限の機能で解決する
**Resource Requirements:** 開発者1名 + AI駆動開発（BMAD + Claude Code）。チーム開発プロセスの検証を兼ねる

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**
- Journey 1（朝の計画→終業確認）: 完全サポート
- Journey 2（後から実績入力）: 完全サポート
- Journey 3（月末集計）: Excel出力のみ（PDF出力はPhase 2）

**Must-Have Capabilities:**
- 2カラムUI（Todoリスト + カレンダー式タイムライン）
- サイドバーナビゲーション（タスク画面 / ダッシュボード画面）
- プロジェクト別Todoグルーピング（折りたたみ式）
- 15分単位のタイムブロック配置・クリック編集
- 1タスクに複数タイムブロック（複数日にまたがる作業対応）
- タスク完了時のタイムブロック編集ロック
- プロジェクトタグ・作業種別タグ（ユーザー自由作成）
- プロジェクト配下タスク作成時のタグ自動付与
- 計画/実績の自動判定ロジック（現在時刻基準）
- 休憩トグル（稼働時間からの自動除外）
- 日次の稼働時間集計表示
- 月次ダッシュボード（日別×プロジェクト別稼働一覧、タグ別集計）
- 月次Excel出力（ワンクリック）
- メールアドレス + パスワード認証

### Phase 2 (Post-MVP)

- PDF出力（振り返りレポート）
- タイムラインブロックのドラッグリサイズ
- ソーシャルログイン（Google等）
- パスワードリセット
- Googleカレンダー連携（会議の自動取り込み）
- パターン学習テンプレート（曜日ごと定例タスク自動配置）
- グラフ可視化（ドーナツ/棒グラフ）

### Phase 3 (Expansion)

- チーム作成・人材管理（マネージャーがチーム全体を俯瞰）
- 時間枠先行モード
- 工数見積もりへの転用

### Risk Mitigation Strategy

**Technical Risks:** タイムラインUIの操作性が鍵。MVPではクリック編集に絞りドラッグ操作は後回しにすることで、複雑なUI実装リスクを軽減
**Market Risks:** 自分自身がユーザーなので、プロダクトマーケットフィットの検証が即座に可能
**Resource Risks:** AI駆動開発により一人でも高速に開発可能。スコープを絞ったMVPで確実にリリースを目指す

## Functional Requirements

### タスク管理

- FR1: ユーザーはプロジェクトを作成・編集・削除できる
- FR2: ユーザーはプロジェクト配下にタスク（Todo）を作成・編集・削除できる
- FR3: ユーザーは「未分類」グループにプロジェクト未定のタスクを作成できる
- FR4: ユーザーはプロジェクトグループを折りたたみ・展開できる
- FR5: ユーザーはタスクに「完了」ステータスを設定できる（完了後はタイムブロックの編集不可）

### タイムライン管理

- FR6: ユーザーはTodoリストからタスクを「今日の予定にする」でタイムラインに配置できる
- FR7: ユーザーはタイムラインのブロックに開始時間・終了時間を15分単位で設定できる
- FR8: ユーザーはタイムラインのブロックをクリックして時間を編集できる
- FR9: ユーザーはタイムラインのブロックを削除できる
- FR10: システムは追加された時間枠が現在時刻より前なら「実績」、以降なら「計画」として自動判定する
- FR11: システムは時間経過に伴い計画を実績に自動変換する（編集されなければそのまま実績に）
- FR12: ユーザーは計画なしで過去の時間枠に直接実績を入力できる
- FR13: ユーザーは1つのタスクに対して複数日にまたがる複数のタイムブロックを作成できる
- FR14: システムはタスクが「完了」に設定された時点で、そのタスクに紐づく全タイムブロックを編集不可にする

### タグ・分類

- FR15: ユーザーはプロジェクトタグを自由に作成・編集・削除できる
- FR16: ユーザーは作業種別タグを自由に作成・編集・削除できる
- FR17: システムはプロジェクト配下でタスク作成時にプロジェクトタグを自動付与する
- FR18: ユーザーはタスクに作業種別タグを手動で選択・付与できる

### 休憩管理

- FR19: ユーザーはタイムライン上に休憩ブロックを配置できる
- FR20: システムは休憩時間を稼働時間の集計から自動除外する

### 集計・ダッシュボード

- FR21: システムは日次の稼働時間をプロジェクト別に自動集計して表示する
- FR22: システムは月次ダッシュボードで日別×プロジェクト別の稼働時間一覧を表示する
- FR23: システムは月次のプロジェクトタグ別合計時間を自動集計して表示する
- FR24: システムは月次の作業種別タグ別合計時間を自動集計して表示する

### レポート出力

- FR25: ユーザーは月次集計データをExcelファイルとしてワンクリックでダウンロードできる

### UI・表示

- FR26: システムはサイドバーナビゲーションで「タスク」画面と「ダッシュボード」画面を切り替えられる
- FR27: システムはタスク画面を2カラムレイアウト（左: Todoリスト、右: カレンダー式タイムライン）で表示する
- FR28: システムはタイムラインをカレンダー式縦時間軸で表示し、ブロックの高さが時間の長さを反映する
- FR29: システムは10:00-19:00の時間範囲をデフォルトでタイムラインに表示する

### 認証・ユーザー管理

- FR30: ユーザーはアカウントを作成してサインアップできる
- FR31: ユーザーはメールアドレスとパスワードでログインできる
- FR32: ユーザーはログアウトできる
- FR33: システムは認証されていないユーザーのアクセスを制限する

## Non-Functional Requirements

### Performance

- タイムラインのブロック操作（追加・クリック編集・削除）: 100ms以内のUI応答
- API応答時間: 500ms以内
- 初期ページロード: 3秒以内
- 月次ダッシュボードの集計表示: 1秒以内

### Security

- パスワードはハッシュ化して保存
- API通信はHTTPS経由
- 認証トークンによるセッション管理
- 認証されていないリクエストの拒否
