# hr-time - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for hr-time, decomposing the requirements from the PRD, UX Design if it exists, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

**タスク管理:**
- FR1: ユーザーはプロジェクトを作成・編集・削除できる
- FR2: ユーザーはプロジェクト配下にタスク（Todo）を作成・編集・削除できる
- FR3: ユーザーは「未分類」グループにプロジェクト未定のタスクを作成できる
- FR4: ユーザーはプロジェクトグループを折りたたみ・展開できる
- FR5: ユーザーはタスクに「完了」ステータスを設定できる（完了後はタイムブロックの編集不可）

**タイムライン管理:**
- FR6: ユーザーはTodoリストからタスクを「今日の予定にする」でタイムラインに配置できる
- FR7: ユーザーはタイムラインのブロックに開始時間・終了時間を15分単位で設定できる
- FR8: ユーザーはタイムラインのブロックをクリックして時間を編集できる
- FR9: ユーザーはタイムラインのブロックを削除できる
- FR10: システムは追加された時間枠が現在時刻より前なら「実績」、以降なら「計画」として自動判定する
- FR11: システムは時間経過に伴い計画を実績に自動変換する（編集されなければそのまま実績に）
- FR12: ユーザーは計画なしで過去の時間枠に直接実績を入力できる
- FR13: ユーザーは1つのタスクに対して複数日にまたがる複数のタイムブロックを作成できる
- FR14: システムはタスクが「完了」に設定された時点で、そのタスクに紐づく全タイムブロックを編集不可にする

**タグ・分類:**
- FR15: ユーザーはプロジェクトタグを自由に作成・編集・削除できる
- FR16: ユーザーは作業種別タグを自由に作成・編集・削除できる
- FR17: システムはプロジェクト配下でタスク作成時にプロジェクトタグを自動付与する
- FR18: ユーザーはタスクに作業種別タグを手動で選択・付与できる

**休憩管理:**
- FR19: ユーザーはタイムライン上に休憩ブロックを配置できる
- FR20: システムは休憩時間を稼働時間の集計から自動除外する

**集計・ダッシュボード:**
- FR21: システムは日次の稼働時間をプロジェクト別に自動集計して表示する
- FR22: システムは月次ダッシュボードで日別×プロジェクト別の稼働時間一覧を表示する
- FR23: システムは月次のプロジェクトタグ別合計時間を自動集計して表示する
- FR24: システムは月次の作業種別タグ別合計時間を自動集計して表示する

**レポート出力:**
- FR25: ユーザーは月次集計データをExcelファイルとしてワンクリックでダウンロードできる

**UI・表示:**
- FR26: システムはサイドバーナビゲーションで「タスク」画面と「ダッシュボード」画面を切り替えられる
- FR27: システムはタスク画面を2カラムレイアウト（左: Todoリスト、右: カレンダー式タイムライン）で表示する
- FR28: システムはタイムラインをカレンダー式縦時間軸で表示し、ブロックの高さが時間の長さを反映する
- FR29: システムは10:00-19:00の時間範囲をデフォルトでタイムラインに表示する

**認証・ユーザー管理:**
- FR30: ユーザーはアカウントを作成してサインアップできる
- FR31: ユーザーはメールアドレスとパスワードでログインできる
- FR32: ユーザーはログアウトできる
- FR33: システムは認証されていないユーザーのアクセスを制限する

### NonFunctional Requirements

**パフォーマンス:**
- NFR1: タイムラインのブロック操作（追加・クリック編集・削除）: 100ms以内のUI応答
- NFR2: API応答時間: 500ms以内
- NFR3: 初期ページロード: 3秒以内
- NFR4: 月次ダッシュボードの集計表示: 1秒以内

**セキュリティ:**
- NFR5: パスワードはハッシュ化して保存
- NFR6: API通信はHTTPS経由
- NFR7: 認証トークンによるセッション管理
- NFR8: 認証されていないリクエストの拒否

### Additional Requirements

- スターターテンプレート: pnpm workspacesモノレポ（create vite + create hono）で初期化。Epic 1 Story 1でプロジェクト基盤をセットアップ
- Hono RPCによる型安全API通信（バックエンドのルート定義からフロントエンドに型が自動伝播）
- Prisma 7.x + PostgreSQL（Railwayホスティング）のデータ層
- Better Auth 1.5.xによるメール+パスワード認証（セッション管理ビルトイン）
- Feature-basedディレクトリ構造（frontend）+ レイヤード構造（backend: routes → services → repositories → Prisma → DB）
- sharedパッケージでzodスキーマ、Prisma型再エクスポート、エラーコード定数を一元管理
- Luxonによる日時処理（API通信: ISO 8601 UTC、UI表示: JST Asia/Tokyo変換）
- ExcelJSによるサーバーサイドExcel生成（クライアントバンドルサイズ抑制のため）
- husky + lint-staged（biome check + tsc --noEmit）によるコミット時品質チェック
- GitHub Actions（biome check + tsc --noEmit + vitest run + ビルドチェック）によるPR時CI
- Railway 1サービスデプロイ（frontend build → backend/public/にコピー → Honoが静的配信+API両方担当。同一ドメインでCORS不要）
- エラー回復戦略: DB確定後リフレッシュ方式（失敗時もデータ不整合なし）、Error Toast（手動クローズ必須）、401検知→ログイン画面リダイレクト、API呼び出し中のボタンdisabled+スピナーで二重送信防止

### UX Design Requirements

**デザインシステム:**
- UX-DR1: Clean Minimalデザイン方向 — 青みがかった白ベース（slate-50系）+ 水色アクセント（sky-400〜sky-500）、shadcn/uiデフォルトスタイルに近い実装
- UX-DR2: カラーシステム — プロジェクト別パステルカラーバリエーション。計画ブロック: パステル+透明度+破線ボーダー、実績ブロック: パステル不透明+実線ボーダー、休憩ブロック: ニュートラルグレー。セマンティックカラー（Success緑/Warning アンバー/Error赤）
- UX-DR3: タイポグラフィ — システムフォントスタックまたはInter/Noto Sans JP。カジュアルで親しみやすいトーン。本文14-16px、補助テキスト12-13px、行間1.5-1.7
- UX-DR4: スペーシング — 8pxベースユニット、ゆったり余白志向、コンポーネント間の十分な間隔で圧迫感排除

**カスタムコンポーネント:**
- UX-DR5: TimelineGrid — 縦時間軸8:00〜22:00、1時間刻み目盛り、控えめなグリッドライン、空白エリアクリック→ポップオーバー表示（「タスクを追加」/「休憩を追加」）、クリック時間帯を開始時間としてプリセット
- UX-DR6: TimeBlock — 5状態管理（計画: パステル+透明度+破線、実績: パステル不透明+実線、ホバー: 微小右方向シフト、選択中: ポップオーバー表示、編集ロック: グレーアウト+編集不可）。高さが実際の時間に比例。左ボーダーにプロジェクトカラー。コンテンツ: タスク名、プロジェクト名、タグ、時間範囲
- UX-DR7: BreakBlock — ニュートラルグレー背景、「休憩」ラベル+時間範囲表示、タスクブロックと視覚的に区別、クリック→ポップオーバーで時間編集 or 削除
- UX-DR8: SelectCreate — Notion風セレクト/クリエイト。入力フィールド（検索/フィルタ兼用）+ 既存項目ドロップダウン + 一致なし時「"入力値" を作成」オプション。プロジェクト選択・タグ選択で使用
- UX-DR9: ProjectGroup — 折りたたみ式グルーピング。プロジェクトカラードット+プロジェクト名+折りたたみアイコン。配下タスクリスト+「+」アイコン（タスク追加時タグ自動付与）。高さ固定スクロールコンテナ内に配置
- UX-DR10: DailySummaryBar — タイムライン下部に常時表示。プロジェクトカラードット+プロジェクト名+稼働時間（「時間」表記）。タイムブロック編集に連動して即時更新

**インタラクションパターン:**
- UX-DR11: タイムライン配置の2導線 — ルートA: Todoリストから直接「今日の予定に追加」→時間選択（最速2クリック）。ルートB: タスククリック→詳細モーダル→確認後「今日の予定に追加」→時間選択
- UX-DR12: TimeEditPopover — タイムブロッククリック→ポップオーバーで開始/終了時間を15分単位ドロップダウンで編集+削除ボタン。外クリックで閉じる
- UX-DR13: 空白クリックポップオーバー — タイムラインの空白クリック時、クリック時間帯をプリセットして「タスクを追加」/「休憩を追加」の選択肢表示
- UX-DR14: モーダルパターン — タスク詳細モーダル（閲覧+「今日の予定に追加」+「編集」ボタン）、タスク編集モーダル（名前/メモ/プロジェクト/タグ/時間変更）、タスク追加モーダル（名前+プロジェクトSelectCreate）。背景オーバーレイ、ESC/外クリックで閉じる

**UIパターン:**
- UX-DR15: ボタン階層 — Primary（水色背景+白テキスト）、Secondary（白背景+水色ボーダー）、Ghost（背景なし+テキストのみ）、Destructive（薄赤背景+赤テキスト）、Brand（Excel出力=#217346）。1画面にPrimary最大1つ
- UX-DR16: Toastフィードバック — Success（3秒自動消去）、Error（手動クローズ必須）、Warning（5秒自動消去）、Info（3秒自動消去）。画面右上表示、最大3件スタック
- UX-DR17: サイドバーナビゲーション — アイコンベース、開閉式（閉56px/開200px）、デフォルト閉じ状態、ホバーでツールチップ、アクティブ状態は水色背景+アイコン色変更
- UX-DR18: 空の状態パターン — Todoリスト初回「タスクを追加して始めましょう」（+ボタンハイライト）、タイムライン「今日の予定はまだありません」、ダッシュボード「まだデータがありません」（タスク画面への導線）

**品質要件:**
- UX-DR19: フォームバリデーション — リアルタイムバリデーション、エラーはフィールド直下赤テキスト、送信ボタンはバリデーション通過後のみアクティブ、フォーカス時水色アウトライン
- UX-DR20: ローディングパターン — スケルトンUI（全画面遷移時）、ボタンdisabled+lucide-react Loader2（animate-spin）（API呼び出し中）、タイムブロック操作はpending状態をUIに表示
- UX-DR21: アクセシビリティ — WCAG Level A準拠、セマンティックHTML、キーボード操作（Tab/Shift+Tab/Enter/Space/Esc）、フォーカスインジケーター（水色アウトライン）、テキストコントラスト比4.5:1以上、ARIA属性（aria-modal/aria-expanded/aria-label）
- UX-DR22: タイムライン表示範囲 — 8:00〜22:00表示（PRDデフォルト10:00-19:00から拡張）、時間比例の縦幅、合計時間は「4.5時間」形式の日本語表記

## Epic 1: 認証とアプリ基盤

ユーザーがアカウントを作成し、ログインしてアプリの基本レイアウトを利用できるようになる。モノレポ・DB・認証の技術基盤もここで構築する。

### Story 1.1: プロジェクト基盤セットアップ

As a 開発者,
I want モノレポプロジェクトが全ツールチェーンとともに初期化されている,
So that 一貫したコード品質と型安全性のもとで開発を開始できる.

**Acceptance Criteria:**

**Given** 既存のhr-timeリポジトリ
**When** モノレポセットアップを実行する
**Then** pnpm workspacesで3パッケージ（frontend/backend/shared）が構成される
**And** frontend: React + Vite + TypeScript（create vite react-ts）が初期化される
**And** backend: Hono + Bun（create hono bun）が初期化される
**And** shared: zodスキーマ・型エクスポート用パッケージが初期化される
**And** Biomeがモノレポ全体に適用される
**And** vitest.workspace.tsでテストが統合される
**And** husky + lint-staged（biome check + tsc --noEmit）が設定される
**And** `pnpm dev`でbackend、`pnpm dev:frontend`でfrontendが起動できる
**And** `.env.example`がfrontend/backendに配置される

### Story 1.2: データベースと認証バックエンド

As a ユーザー,
I want アカウント登録・ログイン・ログアウトのAPIが動作する,
So that 安全にアプリを利用できる.

**Acceptance Criteria:**

**Given** バックエンドが起動している
**When** Prismaスキーマを定義してマイグレーションを実行する
**Then** PostgreSQL上にUserテーブル（nameフィールド含む）とBetter Auth必要テーブルが作成される
**And** Better Auth 1.5.xが`/api/auth/*`エンドポイントで動作する

**Given** 未登録のメールアドレス
**When** `/api/auth/sign-up`に表示名+メール+パスワードでリクエストする
**Then** アカウントが作成され、セッションが返される
**And** パスワードはハッシュ化されて保存される（NFR5）

**Given** 登録済みのメールアドレスとパスワード
**When** `/api/auth/sign-in`にリクエストする
**Then** 認証が成功し、セッションが返される

**Given** 認証済みセッション
**When** `/api/auth/sign-out`にリクエストする
**Then** セッションが無効化される

**Given** 認証されていないリクエスト
**When** `/api/*`（auth以外）にアクセスする
**Then** 401エラーが返される（NFR8）

**And** Honoグローバルエラーハンドラーが統一フォーマット`{ success, data/error }`でレスポンスを返す
**And** Honoビルトインloggerミドルウェアが設定される

### Story 1.3: サインアップ・ログインUI

As a ユーザー,
I want ブラウザからアカウント登録とログインができる,
So that アプリにアクセスして利用を開始できる.

**Acceptance Criteria:**

**Given** 未認証状態でアプリにアクセスする
**When** ページが表示される
**Then** ログイン画面が表示される

**Given** ログイン画面
**When** 「サインアップ」リンクをクリックする
**Then** サインアップ画面に遷移する（表示名+メール+パスワード+確認パスワード入力フィールド）

**Given** サインアップ画面で有効な情報を入力する
**When** 「サインアップ」ボタンをクリックする
**Then** アカウントが作成され、タスク画面に遷移する

**Given** サインアップ画面で無効な入力（空欄、パスワード不一致等）
**When** 入力中にバリデーションが発動する
**Then** フィールド直下に赤テキストでエラーが表示される（UX-DR19）
**And** バリデーション通過までサインアップボタンは非アクティブ

**Given** ログイン画面で登録済みのメール+パスワードを入力する
**When** 「ログイン」ボタンをクリックする
**Then** 認証が成功し、タスク画面に遷移する

**Given** ログイン画面で誤ったパスワードを入力する
**When** 「ログイン」ボタンをクリックする
**Then** エラーメッセージが表示され、再入力が促される

**And** Hono RPCクライアント（lib/apiClient.ts）が設定される
**And** TanStack Query（lib/queryClient.ts）がグローバルエラーハンドラー（401→ログインリダイレクト）付きで設定される
**And** react-hook-form + zodでフォームバリデーションが実装される
**And** shadcn/uiが初期化される（Button, Input, Toast等の基本コンポーネント）
**And** Clean Minimalデザイン（slate-50背景、sky-400〜500アクセント）が適用される（UX-DR1）

### Story 1.4: アプリシェルとナビゲーション

As a 認証済みユーザー,
I want サイドバーナビゲーションでアプリ内を移動できる,
So that タスク画面とダッシュボード画面を切り替えて利用できる.

**Acceptance Criteria:**

**Given** 認証済みでアプリにアクセスする
**When** ページが表示される
**Then** 左端にサイドバーナビゲーション（アイコンベース、デフォルト閉じ56px）が表示される（FR26, UX-DR17）
**And** 「タスク」と「ダッシュボード」の2つのナビゲーション項目がある
**And** アクティブ状態は水色背景+アイコン色変更で表示される
**And** ホバーでツールチップが表示される

**Given** サイドバーの開閉トグル

**And** プロジェクト名にはプロジェクトカラードットが付与される

**Given** 月次ダッシュボード
**When** データの集計が実行される
**Then** 集計表示が1秒以内に完了する（NFR4）

**Given** ダッシュボードにデータがない月
**When** 画面が表示される
**Then** 「まだデータがありません」メッセージとタスク画面への導線が表示される（UX-DR18）

**Given** ダッシュボード画面の読み込み中
**When** データ取得中
**Then** スケルトンUIが表示される（UX-DR20）

**And** バックエンドに`/api/dashboard/monthly`エンドポイント（月指定で日別×プロジェクト別の集計データを返す）が実装される
**And** 休憩ブロックは集計から除外される
**And** Luxonを使用してJST基準で日付集計する

### Story 4.2: タグ別月間集計

As a ユーザー,
I want プロジェクトタグ別・作業種別タグ別の月間合計時間を確認できる,
So that どの案件・どの作業種別にどれだけ時間を使ったか把握できる.

**Acceptance Criteria:**

**Given** 月次ダッシュボード画面
**When** プロジェクトタグ別集計セクションが表示される
**Then** プロジェクトタグごとの月間合計稼働時間が一覧表示される（FR23）
**And** プロジェクトカラードット+タグ名+合計時間が表示される
**And** 合計時間が大きい順にソートされている

**Given** 月次ダッシュボード画面
**When** 作業種別タグ別集計セクションが表示される
**Then** 作業種別タグごとの月間合計稼働時間が一覧表示される（FR24）
**And** タグ名+合計時間が表示される
**And** 合計時間が大きい順にソートされている

**When** タグ別集計が表示される

### Story 4.3: Excel出力

As a ユーザー,
I want 月次集計データをExcelファイルとしてワンクリックでダウンロードできる,
So that 提出用レポートを手作業なしで即時取得できる.

**Acceptance Criteria:**

**Given** 月次ダッシュボード画面
**When** 「Excel出力」ボタンが表示される
**Then** Brandカラー（#217346）のボタンが表示される（UX-DR15）

**Given** 「Excel出力」ボタンをクリックする
**When** Excelファイルの生成が開始される
**Then** ボタンがdisabled+スピナー表示になる（UX-DR20）
**And** サーバーサイドでExcelJSによりExcelファイルが生成される（FR25）

**Given** Excelファイルの生成が完了する
**When** ダウンロードが開始される
**Then** `hr-time_YYYY-MM.xlsx`形式のファイル名でダウンロードされる
**And** Success Toastが表示される（UX-DR16）

**Given** ダウンロードされたExcelファイル
**When** ファイルを開く
**Then** シート1: 日別×プロジェクト別稼働時間一覧（Story 4.1と同じマトリクス形式）
**And** シート2: プロジェクトタグ別月間集計
**And** シート3: 作業種別タグ別月間集計
**And** 各シートにヘッダー行、合計行が含まれる
**And** 適切な列幅とセル書式が設定されている

**Given** Excel生成中にエラーが発生する
**When** サーバーからエラーレスポンスが返る
**Then** Error Toast（手動クローズ必須）が表示される（UX-DR16）
**And** ボタンが再度クリック可能な状態に戻る

**And** バックエンドに`/api/dashboard/export`エンドポイント（月指定でExcelバイナリを返す）が実装される
**And** ExcelJSはサーバーサイドのみで使用する（クライアントバンドルサイズ抑制）
